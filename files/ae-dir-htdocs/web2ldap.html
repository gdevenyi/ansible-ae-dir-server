Only the Æ-DIR-specific use-cases are described herein. Check the web page www.web2ldap.de for more details about installing/customizing/using web2ldap.

Access

Directly connect to Æ-DIR provider for maintaining data:
Connect to local web2ldap installation

Notes:

  • Please don't forget to allow the Æ-DIR providers to set cookies in your browser!
  • You must always authenticate yourself because anonymous access is never allowed.
  • In the welcome e-mail you've received a URL with your personal bind-DN which you can use as a bookmark.

General

This section contains general hints about the user interface layout, common errors, etc.

Layout

At top of the displayed web page always the fixed status area is displayed with the following information:

  • LDAP URI of LDAP server you are connected to
  • Identity bound to the LDAP server as friendly name with description if set in the identity's entry
  • distinguished name of current position in the tree as clickable link list (breadcrumb navigation)

There are two menu bars shown:

Main menu
The main menu on the left always shows the same menu items for choosing the main use-cases.
Context menu
The context menu on the right depends on content displayed in the middle area for starting a related use-case.

In the middle area the following content is displayed depending on the use-case:

  • search forms
  • search results
  • single entries
  • various input forms for adding/modifying entries

Links

Different link types have different colors:

  • Menu links in the main menu on the right or the context menu on the left start a new action.
  • Red links in the middle area lead to functionality within the application.
  • Blue links point to external resources (URLs).

Clicking on an item in the main menu or context menu always (re)starts a use-case and therefore will interrupt a prior action state you reached before in the middle area with your input being lost. Therefore if a use-cases requires several input steps you always reach the next step by hitting a button in the middle area.

Messages

Different message types have different border colors:

  • Warning messages with grey border indicate that some functionality is not available but you can probably proceed with your action but with limited features.
  • Error messages with red border indicate that something failed to work and action was aborted/canceled.

Multi-valued attributes

For some attributes you can add/modify more than one value. You can use the button to add another value or to prepare removing a value. Those buttons actually only alter the input form to be submitted later.

Note that all values have to be distinct. So if there's a select list displayed it will not contain the attribute values already present in other attribute value fields.

Checking user input

web2ldap trys to check the user's input and guide the user by displaying informative error messages if possible. But it's up to the Æ-DIR server to enforce all necessary rules.

Also when displaying existing entries the attribute values are checked. Attribute values considered invalid will be displayed as striked. If you see this you should investigate why the value is considered invalid.

Templates

  • For quickly adding new entries there are several LDIF templates available which mainly defines the set of object classes to be used for a new entry.
  • Various HTML templates were configured for certain object classes to display entries or display input forms when adding/modifying templates.
  • When displaying single entries the attributes not shown in the HTML template area are appended as attribute table below the template area.
  • When adding/modifying entries you can switch from template form to table and LDIF input form and back. Your input will be preserved.
  • More details about use of templates can be found on the web2ldap page.

Connect and bind

Roles:

  1. all users

Prerequisites:

  1. Active aeUser entry has been added for user and he/she knows username and password.

Steps:

  1. Choose one of the web2ldap instances
  2. Enter your user name or bind-DN.
  3. Enter the correct password.
  4. Press .

Note that there are three possible forms of specifying your user's bind-DN with or without the zone's name:

  • short form is 'uid=username,ou=ae-dir'
  • just the username is automagically expanded to short form above
  • full DN is 'uid=username,cn=zone,ou=ae-dir' also possible for direct copy & paste

Common errors and their possible causes:

Invalid credentials:
Either you've entered a wrong password or there is no Æ-DIR account entry for the bind DN used.
Invalid DN syntax:
You've used an syntactically invalid / incomplete bind DN.
Inappropriate authentication / anonymous bind disallowed:
You've probably did not enter a bind DN at all or an empty password resulting in anonymous simple bind which is not allowed.
Strong(er) authentication required:
Anonymous access is not allowed at all. But you tried to send a LDAP request while still not bound with your Æ-DIR account.
Session hijacking detected by wrong cookie. Access denied!
In most cases this is caused by Cookies being disabled/blocked in your browser. Make sure your browser accepts Cookies for Æ-DIR's web2ldap sites.
Session hijacking detected. Access denied!
In very rare cases this is due to some of the session cross-checking env vars changing during the session.

Search

When searching bear in mind that you might not see entries due to access control restrictions. The LDAP server does not distinguish whether you're not allowed to read an entry or whether it does not exist or wasn't found by the filter. It simply does not return the result.

Even if results are returned each of the attributes is also subject to access control restrictions. So some attributes might not be visible for you even though they are present and have some effect.

Choose zone

  1. Navigate to top-level LDAP entry ou=ae-dir by clicking the link in the navigation bar at the top.
  2. Select link Down from the main menu on the left.
  3. → The zones are listed.
  4. Select Read in the middle area besides the wanted zone to navigate into the zone.

Search a single user by username

  1. Select link Search from the main menu on the left.
  2. → The basic search form is displayed in the middle area.
  3. Enter the user's ID in the field "Æ-DIR user name" and
  4. Press ⏎ in the input form or klick the button at top of the middle area.
  5. → Search results are displayed if found.
  6. Select Read in the middle area besides the searched user entry to display it.

Search service group

  1. Select link Search from the main menu on the left.
  2. → The basic search form is displayed in the middle area.
  3. In the select field "Type of entry" choose "Æ-DIR srv. group (aeSrvGroup)".
  4. Optionally input part of the service group's name in field "Common name".
  5. Press ⏎ in the input form or klick the button at top of the middle area.
  6. → Search results are displayed if found.
  7. Select Read in the middle area besides the searched service group entry.

Export servers in a service group

  1. Search service group and display it.
  2. Select link All members in the middle area.
  3. → The list of servers in the service group is displayed in the middle area.
  4. Select link Edit filter from the context menu on the right to define the attribute columns to be exported
  5. Fill in the attributes to be exported as columns in the field "Attributes to be read" as comma-separated list like this: "cn,host,aeStatus,description,aeTicketId"
  6. Press ⏎ in the input form or klick the button at top of the middle area.
  7. → Search results are displayed if found.
  8. Select CSV or Excel at the bottom of the page in the select list in the middle area and press button .

Notes about tree:

  • Adding entries in the wrong part of the tree will likely result in malfunction.
  • Please make yourself familiar with the tree structure (DIT) and LDAP schema used and how the different entries are related (see also Æ-DIR docs).
  • In most cases you will simply have to choose the right zone first.

Schema description: aeUser

Roles:

  • People Admin (zone admin for in zone people)
  • Æ Admin

Add new Æ-DIR person

Prerequisites:

  1. Organizational personnel process has to provide enough information.

Steps:

  1. Connect and bind
  2. Select link New entry from the main menu on the left.
  3. In the middle area select Æ-DIR person.
  4. → Input form is displayed in the middle area. (This template already automatically chooses zone people.)
  5. Make sure web2ldap displays this above the submit buttons to ensure you're adding the new entry directly beneath a zone:
    "Superior entry: Æ-DIR zone people"
  6. Fill in the input values:
    1. Leave the suggested value in input field RDN as is.
    2. Enter names and optionally the e-mail address if already known.
    3. Set "Status:" to…
      • "requested" if you're waiting for some more confirmation before activating the person
      • "active" otherwise
    4. You might want to fill in the fields "Intended purpose" and "Jira ticket for last change".
  7. Press ⏎ anywhere in the input form or click on the Add button at top of the middle area to add the new entry.
  8. → Confirmation of added data is shown in the middle area.
  9. Click link Read in the middle area to display the new entry.

Add new Æ-DIR personal account

Prerequisites:

  1. Person entry (aePerson) already exists.

Steps:

  1. Connect and bind
  2. Choose a zone for the new entry.
  3. Select link New entry from the main menu on the left.
  4. In the middle area select Æ-DIR personal user account.
  5. → Input form is displayed in the middle area.
  6. Make sure web2ldap displays this above the submit buttons to ensure you're adding the new entry directly beneath a zone:
    "Superior entry: Æ-DIR zone..."
  7. Fill in the input values:
    1. Leave the suggested value in input field RDN as is.
    2. In the field "Base person:" enter an person name or e-mail address.
    3. Make sure "Status:" is set to "active".
    4. You might want to fill in the fields "Intended purpose", "SSH public key(s)" and "Jira ticket for last change".
    5. Unix user ID (attribute uidNumber) must be set to at least 30000. Don't muck with the suggested value!
    6. Primary group ID (attribute gidNumber) must be set to solely to value 30000 to avoid zone admins referring groups in other zones.
  8. Press ⏎ anywhere in the input form or click on the Add button at top of the middle area to add the new entry.
  9. → Confirmation of added data is shown in the middle area.
  10. Click link Read in the middle area to display the new entry.

Add or modify SSH key for existing Æ-DIR account

Prerequisites:

  1. The authenticity of the SSH public key has to be checked: By some (out-of-band) security measures you must ensure that the public key really belongs to the user and that only the user can use the private key.
  2. Algorithm and key size of public key has to comply to your crypto security policy.

Steps:

  1. Connect and bind
  2. Search and display user entry
  3. Select link Modify from the context menu at the right.
  4. → Input form is displayed in the middle area.
  5. Copy&paste SSH public key into field "SSH public key(s) (sshPublicKey)"
  6. Click on the button in the middle area or just hit ⏎ key add the new entry.
  7. → Modification to entry will be displayed in the middle area (as LDIF change record).

Change group membership

Prerequisites:

Steps:

  1. Connect and bind
  2. Search and display user entry
  3. Select link Groups from the context menu at the right.
  4. → Two multi-select lists are displayed in the middle area:
    1. "Add to..." lists groups you could add a user to
    2. "Remove from..." lists groups you could remove a user from
    3. Make your choice
  5. Press button in the middle area to apply the changes.

Æ-DIR server/host (aeHost)

For this particular use-case you most likely want to use a custom command-line tool or configuration management (like ansible) instead.

Schema description: aeHost

Roles:

  • Setup Admin
  • Zone Admin
  • Æ Admin

Steps:

  1. Connect and bind
  2. Search service group and select the service group for the new entry.
  3. Select link New entry from the main menu on the left.
  4. In the middle area select Æ-DIR server/host.
  5. → Input form is displayed in the middle area.
  6. Fill in the input values:
    1. Leave the suggested value in input field RDN as is.
    2. Make sure web2ldap displays this to ensure you're adding the server entry beneath a service group above the submit buttons::
      "Superior entry: Æ-DIR service group..."
    3. The first DNS label of input value for "common name" (cn) must match the "FQDN of server" (host).
    4. The input value for "FQDN of server" (host) is checked against DNS. So make sure to create an DNS entry before.
  7. Press ⏎ anywhere in the input form or click on the Add button at top of the middle area to add the new entry.
  8. → Confirmation of added data is shown in the middle area.
  9. Click link Read in the middle area to display the new entry.
  10. Select link Password in the context menu on the right.
  11. → The input form for entering new password is displayed.
  12. Just leave the input fields empty and hit button .
  13. → Random password is generated and displayed together with the full bind-DN for the server.

Step-by-step docs are here just for completeness because every sane admin will use the ae-add-zone.py command-line tool.

  • Æ Admin

  1. Connect and bind
  2. Select link New entry from the main menu on the left.
  3. In the middle area select Æ-DIR zone.
  4. → Input form is displayed in the middle area.
  5. Fill in the input values:
    1. Leave the suggested value in input field RDN as is.

You now have to add two special user groups for the roles zone admins and zone auditors of the new zone.

There are two special LDIF templates available for adding the standard role groups for zone admins and zone auditors:

  • "Æ-DIR zone admins group"
  • "Æ-DIR zone auditors group"

Steps (for each group):

  1. Select link New entry from the main menu when displaying the zone entry and choose one of the templates to add the special user group (see also aeGroup).
  2. The group's name is auto-prefilled based on the zone's name.
  3. Also mention the zone name between the single quotes in the input field Intended purpose (description)
  4. Don't forget to add the ticket number also to these group entries.

Schema browser

If you want to find out more about the details of Æ-DIR's subschema you can dig into it by using web2ldap's powerful schema browser which displays:

  • all schema descriptions (attribute types, object classes, LDAP syntaxes, matching rules, DIT content rules, DIT structure rules and name forms)
  • all cross-references including inheritance relationship

You can always reach the schema browser by selecting link Schema from the main menu on the left.

Small arrow links » are displayed along with names of attribute types and object classes which directly lead to this particular schema description from where you can follow the relationships.
